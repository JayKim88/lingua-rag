## v0.3 범위

### Phase 1: LLM-as-Judge 포맷 검증 (핵심)

**목적**: Claude 응답이 `prompts.py::ANSWER_FORMAT` 규칙을 준수하는지 자동 채점

**검증 항목** (ANSWER_FORMAT 기반):

| 규칙 | 체크 항목 |
|------|---------|
| 독일어 bold 완결성 | 독일어 표현이 `**...**`로 완전히 감싸졌는가 (중간에 끊기면 실패) |
| 표 금지 | 응답에 `\|` 기반 markdown 표 없음 |
| 번역 즉시 배치 | 독일어 줄 직후 `→ 한국어`가 따라오는가 (몰아넣기 금지) |
| 예문 구조 | `A: ... → ...` / `B: ... → ...` 형식 준수 |
| 예문 길이 | 독일어 예문 단어 수 ≤ max_example_length (A1=10, A2=15) |
| 팁 포함 | `💡 팁:` 줄 포함 여부 |

**구현 방식**: `scripts/evaluate.py`

```python
# 입력: 실제 Claude 응답 텍스트 (또는 DB의 messages 테이블)
# 출력: 각 규칙별 pass/fail + 종합 점수 (0-100)

# Judge 프롬프트:
# "다음 독일어 튜터 응답이 아래 형식 규칙을 지키는지 평가하라.
#  각 항목에 대해 pass/fail과 이유를 JSON으로 반환하라."
```

**테스트셋**: 단원별 고정 질문 10개 (수동 작성) → `scripts/test_questions.json`

```json
[
  {"unit_id": "A1-1", "question": "sein 동사 현재형 변화를 알려주세요"},
  {"unit_id": "A1-3", "question": "aus와 in 차이를 설명해주세요"},
  ...
]
```

**검증**: 현재 baseline 측정 → 프롬프트 개선 후 재측정 → delta 보고

### Phase 2: User Feedback 수집 (선택적, Phase 1 완료 후)

- 응답 하단 thumbs up/down UI
- `POST /api/feedback` + `message_feedback` DB 테이블
- 목적: Phase 1 eval 결과와 사용자 체감 품질 상관관계 확인

---

### Phase 3: Multi-turn Query Rewriting (선택적)

- 후속 질문("이 문장 다시 설명해줘") → RAG 검색 전 독립 문장으로 재작성
- RAG_ENABLED=True 복원 후 효과 측정

---

## 착수 순서

```
1. scripts/test_questions.json — 단원별 테스트 질문 10개 작성 (1h)
2. scripts/evaluate.py — Judge 호출 + 결과 파싱 + 리포트 출력 (3h)
3. Baseline 실행 → 점수 기록
4. ANSWER_FORMAT 프롬프트 개선 실험 (필요시)
5. Phase 2 User Feedback (별도 v0.3.x 또는 병행)
```
